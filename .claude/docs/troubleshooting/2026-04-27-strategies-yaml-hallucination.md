# Phase 3 subagent fabricated 108 source URLs in strategies.yaml

**WRONG**: spawn a research subagent with a goal like "research 36
investment strategies and fill `docs/strategies/strategies.yaml`,
populate `sources:` for each".

**RIGHT**: never have an LLM (sub-agent or main) generate fact-heavy
structured data from "research".  Either (a) human writes it by hand
with cited sources, or (b) the LLM is reduced to an extractor:
WebFetch real bytes → store in RawStore → LLM extracts/summarises
THESE specific bytes with cited verbatim phrases → mechanical
post-validation that cited phrase exists literally in the blob.

**WHY**:

The Phase 3 subagent generated `sources` blocks for all 36
strategies.  Each entry got 3 URLs like `https://www.investopedia.com/
terms/q/quadruplewitching.asp`.  At UI surfacing time (2026-04-27 user
audit) they were tested:

- Some 404'd outright (URL never existed)
- Some 200'd but the page content didn't actually back the strategy
  (LLM picked a plausible-sounding URL pattern from training, the
  page existed but discussed something tangential)
- A handful happened to be both real and relevant (lucky coincidence)

The LLM had no way to tell the difference between "URL I actually
visited" and "URL that fits the training-data pattern of
authoritative options-education sites".  Saying "be accurate" in the
prompt does not fix this — the failure mode is structural, not a
prompt-tuning issue.

The rest of the YAML body — `starter_step`, `key_risks`,
`typical_win_rate`, `max_loss`, `feasible_at_10k_reason`,
`tax_treatment.notes` — is generated by the same subagent and may
contain similar fabrications.  The 108 URLs are the visible tip; every
numeric claim in those fields is suspect until a fresh audit grounded
in real RawStore bytes confirms it.

## Detection

- All URLs in `docs/strategies/strategies.yaml` collapsed to
  `sources: []` (cleared 2026-04-27).
- `agent.finance.strategies_catalog.PROVENANCE_STATES`:
  `unverified` (default), `partially_verified`, `verified`,
  `rawstore_grounded`.
- All 36 entries default to `unverified`; only an auditor pass
  (Layer 0) can promote to `verified`.

## Fix

Four guard layers, in order of structural strength:

1. **Schema** — strategies_catalog `_normalise_provenance` ensures
   every entry has `provenance.state`; default `unverified`.
2. **Compute filter** — `is_trusted()` is the single gate; only
   `verified` / `rawstore_grounded` enter the L3 calls prompt.
3. **Auditor** — `agent/finance/strategies/auditor.py` runs LLM as
   extractor over RawStore bytes; mechanical post-check that the
   LLM-cited phrase exists literally; can promote `unverified` →
   `verified` automatically.
4. **UI** — every `unverified` entry shows a ⚠ chip; L3 calls that
   reference one show a degraded `unverified-source` chip.
